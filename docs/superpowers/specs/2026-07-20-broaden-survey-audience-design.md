# 扩大量子化学可视化问卷受众设计

日期：2026-07-20

## 目标

将当前 30 题问卷压缩为不超过 20 题，使亲自计算、参与计算设置、与计算人员合作、委托代算以及主要使用计算结果的受访者都能基于自身经历作答。

## 设计原则

- 不要求非计算受访者猜测程序、参数、文件格式或核验方法。
- 先询问受访者在计算工作流中的参与方式，再按其可观察范围分流。
- 所有人回答需求沟通、成果接收、可视化对象、痛点、信任依据、概念价值和采用障碍。
- 仅亲自运行或直接参与设置的人回答程序、输入输出格式及技术核验题。
- PySCF 保留为一项认知与接触变量，不再设置用户和非用户障碍分支。

## 问卷结构

目标为 18 题，预计 6–9 分钟：

1. 背景与参与方式：4 题。
2. 协作、交付与可信度：4 题。
3. 三维场与可视化需求：5 题。
4. 引擎无关工作流概念测试：4 题。
5. 开放反馈：1 题。

关键分流：

- `workflow_role` 选择“亲自运行计算”或“直接参与方法/参数设置”时，显示技术题。
- 合作者、委托方和结果使用者跳过技术题，继续回答共同需求题。
- 技术题即使被跳过，也不影响问卷完成或进入概念测试。

## 保留与合并

- 保留：研究场景、工作流角色、合作/委托方式、成果类型、信任依据、可视化对象、文件格式、痛点、错误后果、概念价值、功能优先级、交付方式、采用障碍、PySCF 认知和开放案例。
- 合并：程序使用与运行方式；可视化工具与频率；最低元数据与最低验证要求。
- 删除：机构类型、经验年限、主程序、计算频率、程序选择矩阵、交叉核验频率、细粒度核验动作、首要格式、PySCF 实际使用及两套障碍分支。

## 数据与分析

- 首要分层变量为 `workflow_role`，区分计算执行者、共同决策者、合作者、委托方和结果使用者。
- 共同题用于比较不同角色对可视化对象、交付内容、信任证据和采用障碍的差异。
- 技术题只在符合条件的子样本中统计，并明确有效样本数。
- 多选题同时报告受访者比例与选择次数；非概率样本不外推总体市场份额。

## 同步与验证

- Markdown 与工作簿保持相同的 18 个题号、变量、题型和分支。
- 检查题号连续、变量唯一、所有分支目标存在，单选/多选标识明确。
- 重新渲染工作簿问卷页，检查内容可读且无截断。
- 提交前扫描公开文件，确保没有本机路径、身份、机构、Zotero 内部 key 或访问凭据。
