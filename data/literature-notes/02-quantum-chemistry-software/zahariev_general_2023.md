## Zotero 阅读笔记

<h1>Survey 定向阅读问题</h1><p><strong>关联模块：</strong> GAMESS、新架构与高性能方法</p><p>阅读时请直接在每个"回答"后补充结论，并记录对应页码、图表或原文位置。</p><h2>阅读问题</h2><h3>问题 1</h3><p>GAMESS 针对 CPU、GPU 和新架构的优化覆盖哪些方法与规模？</p><p><strong>回答：</strong> GAMESS 的 CPU/GPU 优化覆盖广泛：HF SCF 解析 Hessian 的 MPI 并行始于 1990 年代初（Sec. 1）；RI-MP2 和 CCSD(T) 通过密度拟合和碎片化方法降低标度（Sec. 1）；OpenMP 直接 GPU 卸载（Sec. 3.1）成功应用于 HF 和 RI-MP2 及相关积分；cube 分区方案（Sec. 8）递归拆分含分子的初始立方体作为 GPU 卸载基础；EFP/FMO/EFMO 碎片方法采用 MPI ranks + threads + accelerator 混合并行（Sec. 3.2）；Julia 语言接口（Sec. 10）提供高性能脚本化支持。规模方面覆盖从笔记本电脑到大规模并行集群，包括 AMD GPU 支持（Sec. 3.1），CCSD(T) 采用 RI 近似后可用于中等分子。</p><h3>问题 2</h3><p>碎片化、密度拟合等近似需要哪些额外元数据才能解释可视化结果？</p><p><strong>回答：</strong> 碎片化方法（EFP, FMO, EFMO）产生的能量分解（Sec. 3.2）需要以下元数据：(1) 基组信息和占据 LMO 及其 Fock 矩阵和质心坐标（Sec. 2.2）；(2) 电荷转移贡献需要占据和虚 LMO 信息；(3) 密度拟合/RI 近似（Sec. 2.4.1）需要辅助基组定义和 RI 误差评估；(4) EFP 中非刚性内部自由度需额外的解析梯度信息（Sec. 3.2）；(5) 多层 QM-EFP 实现需要各层边界描述；(6) cube 分区（Sec. 8）用于 GPU 卸载时需空间网格元数据。</p><h3>问题 3</h3><p>传统输入输出与新高性能模块之间是否存在不一致的解析或部署门槛？</p><p><strong>回答：</strong> 存在以下不一致：(1) GPU 卸载（OpenMP）仅覆盖 HF 和 RI-MP2 等部分方法（Sec. 3.1），CCSD(T) 等高阶方法仍以 CPU 为主；(2) 碎片化方法（EFP/FMO/EFMO）的输入相对复杂，需预先定义碎片及其相互作用参数；(3) CUDA-GPU 版本依赖特定硬件（NVIDIA），AMD GPU 的支持仍在发展中；(4) Julia 接口（Sec. 10）降低了部署门槛但处于初步阶段；(5) 传统文本输入格式与新的硬件优化模块（如 cube 分区 GPU 卸载）之间缺乏自动化转换工具。</p><h2>阅读后回填</h2><ul><li>对问卷题目的影响：可增加关于"碎片化方法元数据可视化需求"和"GPU/CPU 混合架构对可视化工具输出格式的要求"的题目</li><li>可转化为访谈追问：用户对 GAMESS GPU 加速和 Julia 接口的使用体验</li><li>关键证据页码/图表：Sec. 1 (并行化历史), Sec. 3.1 (GPU OpenMP), Sec. 3.2 (碎片化), Sec. 8 (cube 分区), Sec. 10 (Julia 接口)</li><li>仍未解决的问题：GAMESS 是否计划支持通用可视化输出格式（Molden/Cube/XML）？</li></ul><h2>综合摘要</h2><p>本文（Zahariev et al., J. Chem. Theory Comput. 19, 7031-7055, 2023）展示了 GAMESS 在新架构上的系统性优化：OpenMP 直接 GPU 卸载（Sec. 3.1）用于 HF 和 RI-MP2，cube 分区策略（Sec. 8）支持 GPU 加速，EFP/FMO/EFMO 碎片化方法（Sec. 3.2）实现大规模并行（MPI+OpenMP+Accelerator），以及 Julia 接口（Sec. 10）提供高性能脚本化入口。密度拟合（RI）和碎片化近似虽降低了高阶相关方法的标度，但也引入额外的元数据需求。GAMESS 在从传统 I/O 到现代混合架构的演进中，不同模块间的部署门槛和输出格式不一致仍是独立可视化工具集成的潜在障碍。</p>
